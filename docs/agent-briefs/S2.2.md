# S2.2 тАФ CLI: `krwn module pack` + `sign` (Ed25519)

**Stage:** 2 ┬╖ **Size:** M (2 days) ┬╖ **Blocked by:** S2.1 ┬╖
**Blocks:** S2.4

## Goal

Implement the publisher-side tooling: generate Ed25519 keypairs, produce
`.krwn` archives from a module source directory, and sign them with a
detached `.sig`. Uses only `node:crypto` and common stdlib тАФ no new heavy
dependencies.

## Why this matters

Publishers (including us, starting with our own first-party modules in
S2.5) need a reliable, scriptable way to ship signed packages. The CLI is
the ergonomic surface that determines whether other people will actually
publish modules or just give up.

## Starting state

- `cli/bin/krwn.mjs` тАФ CLI entry point.
- `cli/src/` тАФ command implementations.
- Existing commands: `login`, `module validate`, `module install` (installs
  from a local path, unsigned), `vertical`, `invite`, `backup`, `token`,
  `status`. Reference `cli/src/commands/module/validate.ts` for a similar
  shape to follow.
- `packages/sdk/src/manifest.ts` тАФ manifest loader.
- `docs/MODULE_PACKAGING.md` from S2.1 тАФ the spec. Required reading before
  writing any code.

## Do this

1. `krwn key generate [--out <path>]`
   - Produces an Ed25519 keypair via `crypto.generateKeyPairSync('ed25519')`.
   - Writes the **private key** to `<path>/krwn-publisher.key` (PEM, 0600).
   - Writes the **public key** to `<path>/krwn-publisher.pub` (PEM).
   - Prints a suggested `publisherKeyId` (e.g. first 16 hex chars of
     SHA-256 of the public key DER) that the publisher submits to the
     registry.
2. `krwn module pack <src-dir> [--out <file.krwn>]`
   - Validates the manifest at `<src-dir>/krwn.module.json`.
   - Enforces the layout and size limits from S2.1's spec.
   - Produces a gzipped tarball at `<file.krwn>` (default:
     `<slug>-<version>.krwn` in cwd).
   - Exit 0 on success, non-zero with a clear error on any layout
     violation.
3. `krwn module sign <file.krwn> --key <private-key-path> --key-id <id>`
   - Loads the private key, computes SHA-256 of the archive bytes,
     wraps the envelope per S2.1, signs, writes `<file.krwn>.sig`.
   - Verifies its own signature before exiting (self-test).
4. A round-trip unit test in `cli/src/__tests__/module-pack-sign.test.ts`:
   - Generate keypair in a tmp dir.
   - Pack the existing `fixtures/modules/example-good/` directory.
   - Sign the archive.
   - Re-load the public key and verify the signature from scratch using
     `crypto.verify`, ignoring the CLI's own self-test path.
5. Update `docs/CLI.md` with the three new commands and a short example
   flow. Match the tone of existing command docs.

## Definition of done

- [ ] `krwn key generate`, `krwn module pack`, `krwn module sign` all
      work end-to-end.
- [ ] Round-trip test passes on a fresh clone.
- [ ] Private keys are written with mode 0600 (`fs.chmodSync`).
- [ ] `docs/CLI.md` updated.
- [ ] No new dependencies in `cli/package.json`. Stick to `node:crypto`,
      `node:zlib`, and an existing tar library if one is already in the
      tree (check `cli/package.json` first тАФ if tar isn't there, add
      `tar` which is the standard and small).
- [ ] `ROADMAP.md ┬з5` gains a `[~]` marker on the Signed Modules line
      noting pack/sign is done; the install side remains pending
      until S2.4.

## Don't do

- Don't implement `krwn module install <file.krwn>` signature verification
  here. That's S2.4 тАФ deliberately split so the verifier gets a clean PR
  with the security surface in one place.
- Don't generate RSA or ECDSA keys. Ed25519 only. It's small, fast, and
  the stdlib supports it natively.
- Don't embed the private key in any output other than the file on disk.
  No stdout dumps, no logs, nothing.
- Don't skip the self-test in `sign`. Round-tripping at sign time catches
  canonicalization bugs before they hit the verifier.

## Pitfalls

- `crypto.generateKeyPairSync('ed25519')` returns `KeyObject`s. Use their
  `.export({ type: 'pkcs8', format: 'pem' })` for private and
  `.export({ type: 'spki', format: 'pem' })` for public. Don't try to
  serialize them manually.
- Gzipped tar is not deterministic by default (mtime, owner, etc.). Set
  `portable: true` and explicit mtime on the tar entries so that
  `pack <dir> тЖТ archive1` and `pack <same dir> тЖТ archive2` produce
  byte-identical output. This will be load-bearing for reproducible
  builds later.
- Windows file permissions: `fs.chmodSync(..., 0o600)` is a no-op on
  Windows. Warn the user if platform is `win32` and print guidance to
  protect the key file via Windows ACLs or DPAPI. Don't fail тАФ degrade
  gracefully.
