# S2.5 — Dogfood: sign all first-party modules

**Stage:** 2 (final) · **Size:** M (1–2 days) · **Blocked by:** S2.4 ·
**Blocks:** S3.1

## Goal

Re-package `core.chat`, `core.wallet`, `core.governance`, and
`core.tasks` as signed `.krwn` bundles using the KrwnOS core publisher
key. Prove the full publishing pipeline works end-to-end on our own
code before inviting third parties. Wire this into CI so every build
exercises the pipeline.

## Why this matters

The marketplace (Stage 3) will ship modules the Sovereign didn't write.
If our own first-party modules can't even round-trip through the
signing pipeline, nothing else can. Dogfooding here catches a whole
class of "but it worked locally" bugs before a third party hits them.

## Starting state

- S2.2 — `krwn key generate`, `krwn module pack`, `krwn module sign`
  implemented.
- S2.3 — publisher registry with KrwnOS Core key seeded.
- S2.4 — verifier implemented on CLI + server.
- `src/modules/chat`, `src/modules/wallet`, `src/modules/governance`,
  `src/modules/tasks` — currently bundled first-party modules, not
  shipped as `.krwn` packages.
- `fixtures/modules/` — fixtures from S2.4.

## Do this

1. For each first-party module, ensure a valid `krwn.module.json` lives
   at the module root alongside the code. Some may already have
   manifests; check each. Fill in `publisherKeyId` referencing the
   core publisher.
2. Add a script `scripts/build-first-party-modules.mjs` that:
   - Iterates the four modules.
   - Runs `krwn module pack` on each into a build directory (e.g.
     `build/modules/`).
   - Runs `krwn module sign` on each.
   - Fails loudly if any step errors.
3. Add a CI step in `.github/workflows/ci.yml` that runs the script
   after `next build` and before e2e. The signing step needs access
   to the private key; do NOT commit it. Use a GitHub Actions secret
   (`KRWN_PUBLISHER_PRIVATE_KEY`) decoded into a tmp file.
4. Add a CI step that then runs `krwn module install` against a
   scratch Postgres for each signed bundle and verifies it installs,
   as an end-to-end sanity check. Use the same Postgres + Redis
   services the e2e suite already starts.
5. Update `fixtures/modules/` to include one "good-signed" and one
   "bad-signed" first-party fixture, generated by the above script and
   re-used by S2.4's integration test (so the fixtures aren't duplicated
   between CLI-side and server-side tests).
6. Update `docs/MODULE_GUIDE.md` with a "Publishing" section covering:
   - How to generate a publisher key.
   - How to request inclusion in `publishers.json`.
   - How to pack, sign, and test an install.
   - Expected manifest fields.
7. Update `ROADMAP.md §5` to mark Signed Modules fully closed with a
   commit ref. Stage 3 (marketplace) is unblocked after this merges.

## Definition of done

- [ ] All four first-party modules have valid `krwn.module.json`
      manifests.
- [ ] `scripts/build-first-party-modules.mjs` packs + signs all four
      deterministically.
- [ ] CI runs the script on every push and fails on any error.
- [ ] CI installs each signed bundle against a scratch Postgres and
      verifies the `InstalledModule` row appears.
- [ ] `fixtures/modules/` has signed + tampered fixtures reused across
      tests.
- [ ] `docs/MODULE_GUIDE.md` "Publishing" section complete.
- [ ] `ROADMAP.md §5` Signed Modules line fully `[x]`.

## Don't do

- Don't check the private key into the repo or into fixtures. It lives
  in a GitHub Actions secret and on the Sovereign's machine, nowhere
  else.
- Don't make the first-party build use a different pack format from
  third-party. Same pipeline, same verifier, same everything.
- Don't let the bundled-vs-installed question drift: these modules are
  still bundled via `registry.register()` in `src/modules/index.ts`.
  Signing them proves the pipeline works; it doesn't change how they
  load in the running process. That's a larger architectural change
  for Horizon 3/4 to consider separately.

## Pitfalls

- Reproducibility: if `npm run build` of a module produces different
  bytes on two machines, the signature becomes unstable. Lock down
  `NODE_ENV`, module-bundler timestamps, and file ordering. The
  `portable: true` flag on tar from S2.2 matters here.
- CI secret leakage: never `echo "$KRWN_PUBLISHER_PRIVATE_KEY"` in a
  log line, even as part of an error message. Use GitHub's secret
  masking, and write the key to a tmp file with mode 0600 for the
  duration of the signing step only.
- Manifest hygiene: module versions must bump when code changes. If
  you sign two different builds with the same `version` field, it
  invalidates cache assumptions downstream. Tie module version to
  KrwnOS platform version for first-party modules to start.
