# S1.4 — Reconcile `CHANGELOG.md` vs `ROADMAP.md §9 Done`

**Stage:** 1 · **Size:** S (2–3 hours) · **Blocked by:** S1.3 ·
**Blocks:** S1.5

## Goal

Pick exactly one source of truth for scope/delivery tracking and exactly
one source of truth for user-facing release notes, document the split
clearly, update the cursor rule, and trim the current duplication.

## Why this matters

On 2026-04-22 the project gained `CHANGELOG.md` plus
`.cursor/rules/changelog.mdc` — but `docs/ROADMAP.md §9 Done` has been the
canonical delivery log since the project started, referenced by §0 of the
roadmap as the living contract. Right now both documents claim to be
authoritative, and agents following the cursor rule will double-write.
Without reconciliation, the contract gets noisier with every merge.

## Starting state

- `CHANGELOG.md` at the repo root, with four 2026-04-22 entries in
  `[Unreleased] → Added`.
- `.cursor/rules/changelog.mdc` instructing agents to update
  `CHANGELOG.md` on every meaningful commit.
- `docs/ROADMAP.md §0` — the living-contract rules (2026-04-22 state).
- `docs/ROADMAP.md §9 Done` — the existing delivery log, organized by
  horizon, with commit refs.

## Recommended split (adopt unless you have a strong reason not to)

- **`docs/ROADMAP.md §9 Done`** = authoritative scope/delivery log.
  One entry per closed roadmap item, organized by horizon, with date +
  commit/PR ref. Agents write here immediately on merge (already the rule
  in §0).
- **`CHANGELOG.md`** = user-facing release notes. "Keep a Changelog"
  format. Populated **at tag time** (not per-commit) by synthesizing the
  user-visible items from ROADMAP §9 Done since the last tag. Internal
  refactors, test additions, and infra-only work do not appear here.

## Do this

1. Amend `docs/ROADMAP.md §0` with an explicit subsection: "Scope log vs
   release notes." Name the two documents, state the rule: §9 is written
   per PR, `CHANGELOG.md` is written per tag.
2. Rewrite `.cursor/rules/changelog.mdc` to match. Concretely: it should
   instruct agents to (a) always update ROADMAP §9 in the same PR as the
   change, and (b) only add a `CHANGELOG.md` entry if the change is
   **user-visible** (new feature a Sovereign would notice, a migration,
   or a breaking change). Internal plumbing does not go to `CHANGELOG.md`.
3. Trim `CHANGELOG.md [Unreleased]`: of the four 2026-04-22 entries, the
   DB sandboxing, `ModuleContext.secrets`, and the rule-file creation are
   *not* user-visible — move them out (into ROADMAP §9 if not already
   there). Keep only `core.tasks` since that's a new feature a Sovereign
   sees. Keep the `[Unreleased]` section; it will accumulate until the
   next tag.
4. Update `docs/MODULE_GUIDE.md` to mention this split briefly, since
   module authors publishing through the future marketplace (Stage 3)
   will ask where to write their release notes.

## Definition of done

- [ ] `docs/ROADMAP.md §0` has a "Scope log vs release notes" subsection.
- [ ] `.cursor/rules/changelog.mdc` is consistent with §0.
- [ ] `CHANGELOG.md [Unreleased]` contains only user-visible items.
- [ ] `docs/MODULE_GUIDE.md` references the split.
- [ ] No test or build impact — this is docs-only.

## Don't do

- Don't delete `CHANGELOG.md` or `.cursor/rules/changelog.mdc`. Both stay;
  they just have sharper scopes.
- Don't retroactively rewrite `ROADMAP.md §9` history. Old entries stay as
  they are — the rule applies going forward.
- Don't introduce a third doc. Two is already one too many for some
  projects; a third would be absurd.

## Pitfalls

- `.cursor/rules/changelog.mdc` is YAML-frontmattered Markdown. Preserve
  the frontmatter (`description`, `globs`) when rewriting.
- "User-visible" is fuzzy. Rule of thumb: if a Sovereign would notice
  the change by looking at the dashboard, clicking a new page, or
  running a new CLI command, it belongs in `CHANGELOG.md`. Bug fixes in
  hot paths and security improvements also count. Pure test additions,
  lint fixes, and internal refactors do not.
