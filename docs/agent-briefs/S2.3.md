# S2.3 ‚Äî Publisher key registry (v0: static JSON)

**Stage:** 2 ¬∑ **Size:** S (half day) ¬∑ **Blocked by:** S2.1 ¬∑
**Can run parallel with S2.2** ¬∑ **Blocks:** S2.4

## Goal

Ship a minimal, static publisher-key registry inside `@krwnos/sdk` so that
signature verification (S2.4) has a trusted source to resolve
`publisherKeyId` ‚Üí public key. Seeded with the KrwnOS core publisher key.

## Why this matters

Without a registry, signing is theater ‚Äî anyone could sign anything. The
v0 is a static JSON file baked into the SDK; it gives us a trust anchor
for first-party modules (S2.5) and a clear upgrade path to a real
registry service later (Horizon 4). Deliberately dumb, deliberately
small.

## Starting state

- `packages/sdk/src/` ‚Äî SDK source.
- `packages/sdk/src/index.ts` ‚Äî public exports.
- No publisher-key file exists yet.
- S2.1 has defined `publisherKeyId` format (first 16 hex chars of
  SHA-256 of SPKI-DER public key, or whatever the spec actually says ‚Äî
  use that verbatim).

## Do this

1. Create `packages/sdk/src/publishers.json` with the following shape:
   ```json
   {
     "version": 1,
     "publishers": {
       "<keyId>": {
         "name": "KrwnOS Core",
         "publicKey": "-----BEGIN PUBLIC KEY-----\n...\n-----END PUBLIC KEY-----\n",
         "website": "https://krwnos.com",
         "addedAt": "2026-04-YY"
       }
     }
   }
   ```
   Seed with exactly one entry: the KrwnOS core publisher key. The
   Sovereign will generate this key via `krwn key generate` (S2.2) and
   commit the public half here. Private key does not go in the repo.
2. Add a loader API in `packages/sdk/src/publishers.ts`:
   - `loadPublishers(): PublisherRegistry` ‚Äî reads the bundled JSON.
   - `resolvePublisher(keyId: string): Publisher | null`.
   - `isKnownPublisher(keyId: string): boolean`.
   Export from `packages/sdk/src/index.ts`.
3. Add unit tests `packages/sdk/src/__tests__/publishers.test.ts` for
   loader behavior (known key, unknown key, malformed JSON rejected).
4. Confirm the JSON file is bundled into the Next.js build. If using
   `import pub from './publishers.json'`, TypeScript's
   `resolveJsonModule` must be enabled (check `packages/sdk/tsconfig.json`).
   Verifier (S2.4) must work offline, with no network calls.
5. Document the registry shape in `docs/MODULE_PACKAGING.md` (extend
   S2.1's work) with a "Publisher Key Registry" section.

## Definition of done

- [ ] `packages/sdk/src/publishers.json` exists with the KrwnOS core
      publisher entry.
- [ ] Loader API exported from `@krwnos/sdk`.
- [ ] Unit tests pass.
- [ ] Offline verify works: `npm run build` + disconnect network + import
      the SDK and call `resolvePublisher` with a known key.
- [ ] `docs/MODULE_PACKAGING.md` extended with registry section.

## Don't do

- Don't implement a fetched/remote registry. This is v0 deliberately.
  The marketplace in Stage 3 may end up serving `publishers.json` over
  HTTPS, but not in this PR.
- Don't store private keys anywhere in the repo. Only public keys.
- Don't add multiple publishers in this PR. Seed with one (KrwnOS Core).
  Additional publishers are added via subsequent PRs with their own
  review process.
- Don't use revocation semantics yet. If a key needs to be revoked, you
  remove the entry and bump `version`. Formal revocation with timestamps
  is a Horizon 4 concern.

## Pitfalls

- JSON imports and tree-shaking: some bundlers strip "unused" JSON. Test
  that the file is actually present in the built SDK (`dist/`), not just
  during dev.
- Monorepo path resolution: `@krwnos/sdk` is a workspace package. The
  loader path must work both in dev (`tsx` direct) and in the built
  artifact. Use `new URL('./publishers.json', import.meta.url)` via
  `fs.readFileSync` if JSON import fails in the ESM build path.
- Do not hand-edit the PEM strings. Have the Sovereign run
  `krwn key generate` (S2.2), then copy the exact contents of
  `krwn-publisher.pub` into the JSON. Line breaks matter.
