# S2.1 — Spec the `.krwn` archive format

**Stage:** 2 (signed modules) · **Size:** S (1 day) ·
**Blocked by:** S1.5 · **Blocks:** S2.2, S2.3, S2.4, S2.5

## Goal

Land a design document — `docs/MODULE_PACKAGING.md` — that specifies the
`.krwn` archive layout, the signing envelope, the canonicalization rules,
and the install flow. Before any code, before any CLI work, before the
registry.

## Why this matters

`.krwn` packages are the contract between module publishers and Sovereign
installs. If the spec is wrong, every downstream piece (CLI pack/sign,
verifier, registry, marketplace) has to change. A one-day spec pass saves
a week of rework. Also: third-party publishers will read this doc forever
— write it once, write it well.

## Starting state

- `packages/sdk/src/schemas/krwn-module-manifest.schema.json` — existing
  manifest JSON Schema (slug, version, permissions, migrations path,
  peerDeps, ui entrypoint, schema name).
- `packages/sdk/src/manifest.ts` — `validateKrwnModuleManifest`.
- `fixtures/modules/example-good/` — an existing fixture used by manifest
  tests. Useful model for what a source layout looks like pre-archive.
- No `.krwn` archive format exists yet. No signing yet.

## Do this

Write `docs/MODULE_PACKAGING.md` covering the following sections. Each
must be concrete enough that the CLI-side agent (S2.2) and the verifier
agent (S2.4) can implement without asking questions.

1. **Archive layout.** What's inside a `.krwn`? Propose: gzipped tarball
   (`.tar.gz` equivalent), top-level files (`krwn.module.json`, `README.md`,
   `LICENSE`), directories (`migrations/`, `dist/` with compiled JS,
   optional `ui/`). Settle on required vs optional paths. Include a
   tree diagram.
2. **Maximum sizes.** Uncompressed and compressed. Reject archives above
   the limit at verify time to prevent decompression bombs. Reasonable
   starting point: 50 MB compressed, 500 MB uncompressed, capped to
   10,000 files.
3. **Signing envelope.** Propose: detached `.sig` file (e.g.
   `finance.krwn` + `finance.krwn.sig`). The signature covers the
   SHA-256 of the archive bytes plus the publisher key id, serialized in
   a deterministic way.
4. **Canonicalization.** What bytes get hashed before signing? Start
   simple: the literal bytes of the tarball + the publisher key id in a
   well-defined envelope. Defer in-manifest canonicalization unless
   there's a concrete reason to add complexity.
5. **Manifest schema updates.** If the package needs new fields
   (`publisherKeyId`, `archiveHash`, `minPlatformVersion`), list them
   here and update `packages/sdk/src/schemas/krwn-module-manifest.schema.json`
   in this same PR. Bump the manifest schema version.
6. **Install flow diagram.** Sovereign invokes
   `krwn module install ./finance.krwn`. Steps: (1) open archive and
   read manifest, (2) resolve `publisherKeyId` against the publisher
   registry, (3) verify signature against archive bytes, (4)
   `validateKrwnModuleManifest`, (5) check `minPlatformVersion`, (6)
   run migrations in the sandboxed schema, (7) register the module via
   `registry.register()`. A failure at any step aborts cleanly with a
   distinct error code.
7. **Error taxonomy.** List the specific error codes the verifier will
   emit: `KRWN_PKG_BAD_ARCHIVE`, `KRWN_PKG_UNKNOWN_PUBLISHER`,
   `KRWN_PKG_BAD_SIGNATURE`, `KRWN_PKG_MANIFEST_INVALID`,
   `KRWN_PKG_VERSION_MISMATCH`, etc. S2.4 implements these.
8. **Backwards compatibility.** State how the format will evolve. Propose:
   the top-level tarball carries a `KRWN_FORMAT_VERSION` file (a single
   integer) so verifiers can refuse archives they don't understand.

## Don't do

- Don't implement the packer or verifier in this PR. Spec only.
- Don't invent a custom archive format (zip, custom TLV, etc.). Tarball
  is a deliberate choice for broad tooling support.
- Don't introduce encryption at rest in the archive. Modules are public
  artifacts; signing proves provenance, not secrecy. Secrets live in
  `ctx.secrets`, not the package.
- Don't pull in a heavy crypto dependency. Ed25519 is in `node:crypto`
  (Node 20+). Stay with stdlib.

## Pitfalls

- "Canonicalization" is where security bugs live. Be explicit about byte
  ordering and avoid any case where the same logical manifest can have
  two different hashes. Hashing the raw tarball bytes (not a
  re-serialization of its contents) avoids this entire class of bug.
- Don't conflate the publisher key with the State operator key. These
  are different trust domains: State operator signs `.well-known/krwnos.json`
  (future Horizon 4); publisher signs `.krwn` packages.
