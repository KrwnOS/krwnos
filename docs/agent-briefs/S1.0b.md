## Decision rules per discrepancy

Work through each discrepancy in turn. For each, the question is:
*does the module's assumption reflect a legitimately useful SDK direction
(extend the SDK), or a mistake (fix the module)?* Per-item decisions
below — apply them verbatim unless you have a concrete reason to deviate,
in which case stop and report.

### 1. `ctx.auth.userId` → **extend the SDK**

Modules need to know who the caller is. Other first-party modules
currently get user identity from route-level helpers rather than
`ModuleContext`, which has been workable but doesn't scale to
third-party modules installed from the marketplace (S3.1). Add:

```ts
// packages/sdk/src/module-contract.ts
export interface ModuleAuth {
  userId: string;
}
export interface ModuleContext {
  // ...existing fields
  auth: ModuleAuth | null;   // null for system-initiated calls (jobs, hooks)
}
```

In `src/core/module-sandbox.ts`, surface `userId` from the
`SandboxedContextOptions` as `auth: userId ? { userId } : null` on the
returned context.

### 2. `ctx.events.emit(...)` → **rewrite the module to use `ctx.bus`**

The sandbox already exposes `ctx.bus` — the canonical event primitive
in the kernel (`EventBus` in `src/core/event-bus.ts`). "events" is a
naming collision. Do NOT add an `events` alias. Rewrite every
`ctx.events.emit(name, payload)` in the tasks module to the actual bus
method (check `EventBus`'s public API — likely `publish(name, payload)`
or `emit(name, payload)` on the `bus` itself). This keeps the SDK
surface small.

### 3. `KrwnError` → **export from `@krwnos/sdk`**

The tasks module imports it; other future modules will want to too.
Add a minimal exported class:

```ts
// packages/sdk/src/errors.ts
export class KrwnError extends Error {
  constructor(public override readonly message: string,
              public readonly code: string) {
    super(message);
    this.name = "KrwnError";
  }
}
```

Export from `packages/sdk/src/index.ts`. If a similar class already
exists inside `src/core/` or elsewhere in the repo, prefer re-exporting
the existing one over creating a second.

### 4. `PermissionsEngine.can(...)` signature → **rewrite the module**

The engine's existing signature is the contract. Other modules
(governance, wallet, chat) depend on it. Open
`src/core/permissions-engine.ts` to see the real signature — typically
something like `can(input: CanInput, permission: PermissionKey): boolean`
where `CanInput` is `{ stateId, userId, nodeId? }`. Rewrite every call
site in `src/modules/tasks/service.ts` to match. Do NOT change the
engine's public signature — that's a cross-cutting change with
unrelated scope.

### 5. Route-level `getAuthenticatedContext` drift → **minimal stub in chat `_context.ts`**

The tasks routes import a function that doesn't exist at
`@/app/api/chat/_context`. S1.2 will extract a proper shared helper —
but until then, the routes need to at least compile.

Add a minimal `getAuthenticatedContext(req)` to
`src/app/api/chat/_context.ts` that returns a `ModuleContext`-shaped
object populated from the request's auth (follow the pattern other
existing chat routes use to verify the bearer token). Do NOT move it
to a shared location — that's explicitly S1.2's job. Leave a `// TODO:
S1.2 — extract to @/app/api/_shared/auth-context` comment so the next
agent finds it immediately.

Also fix `import prisma from "@/lib/prisma"` in the route handlers —
the module has a named export (or no default export). Check
`src/lib/prisma.ts` and conform. Typical shape is
`import { prisma } from "@/lib/prisma"`.

### 6. `KanbanBoard.tsx` `ITask` mismatch → **fix the type**

Small type correction. Read the error, fix locally. If the fix is
non-obvious, stop and report.

### 7. `module-sandbox.test.ts` — 4 `SandboxedContextOptions` errors → **fix the test to match the implementation**

These predate the tasks work. Align the test fixtures with the current
`SandboxedContextOptions` shape in `src/core/module-sandbox.ts`.

### 8. `vitest.config.mts.timestamp-*.mjs` untracked → **add to `.gitignore`**

Add `vitest.config.mts.timestamp-*.mjs` (or a broader
`*.timestamp-*.mjs` glob) to the root `.gitignore`. Do not commit the
file itself.

## Pitfalls

- Extending `ModuleContext` means you must also update the harness
  (`packages/sdk/src/harness.ts`) and any tests that construct a
  `ModuleContext` by hand, or TypeScript will complain. The
  `module-sandbox.test.ts` fix in decision 7 may partially overlap.
- If extending `ModuleContext` breaks the existing wallet / chat /
  governance modules (because they may be constructing or destructuring
  `ModuleContext`), stop. That's a signal the extension is too invasive
  and decision 1 should be reconsidered as "fix the module instead."
  Report and wait for direction.
- `npm install` on Windows can leave dangling symlinks under
  `node_modules/@krwnos/` from workspace packages. Harmless, but if
  `npm run typecheck` fails with "cannot find module @krwnos/sdk",
  re-run `npm install` first.
