# S1.3 — Wire `core.tasks` through the module sandbox

**Stage:** 1 · **Size:** M (1 day) · **Blocked by:** S1.2 · **Blocks:** S1.4

## Goal

Make `core.tasks` consume the `ModuleContext` sandbox contract that shipped
on 2026-04-22, instead of reaching into the global Prisma client. Tables for
tasks must live in their own schema (`krwn_tasks_<stateIdPrefix>`), and
secrets (if any) must flow through `ctx.secrets.get()`.

## Why this matters

The whole point of the sandbox work (`src/core/module-sandbox.ts`) is that
untrusted modules cannot corrupt core tables. `core.tasks` shipped before
the sandbox existed and still imports the global prisma client directly —
which means it's both (a) violating the contract we just established and
(b) failing to exercise that contract on our own code. If first-party
modules don't dogfood the sandbox, third-party modules installed from the
marketplace (Stage 3) won't either.

## Starting state

- `src/core/module-sandbox.ts` — defines `createSandboxedModuleContext`
  with `db`, `secrets`, `bus`, `logger`, `permissions`. Read this first.
- `src/core/__tests__/module-sandbox.test.ts` — existing coverage for the
  sandbox itself.
- `packages/sdk/src/module-contract.ts` — the public `ModuleContext` shape
  modules see.
- `packages/sdk/src/prisma-per-schema.ts` — helper for scoped schemas.
- `src/modules/tasks/repo.ts` currently imports
  `import prisma from "@/lib/prisma"` (or similar) — this is the violation
  to remove.

## Do this

1. Change `TasksRepository`'s constructor to take a `ModuleDatabase`
   (from `@krwnos/sdk`), not the raw prisma client. Update its methods to
   use `ctx.db.transaction(...)` for writes and scoped reads.
2. In `src/modules/tasks/index.ts`, register the module so it receives a
   sandboxed context on init. The schema name convention is
   `krwn_tasks_<stateIdPrefix>` — lift this from the sandbox helper,
   don't hard-code.
3. Update all `src/app/api/tasks/**/route.ts` handlers to construct the
   sandboxed context (via the same path chat/wallet/governance use, once
   S1.2 has unified auth), then hand that to `TasksService`. No route
   should import `@/lib/prisma` to build the tasks service.
4. Add a migration (or amend the pending one) that creates the tasks
   schema and tables in the sandboxed location. Cross-check
   `prisma/schema.prisma` against the sandbox schema naming convention.
5. Add a regression test in `src/modules/tasks/__tests__/sandbox.test.ts`
   that fails if anything in `src/modules/tasks/**` imports from
   `@/lib/prisma`. A simple grep-based test or a custom eslint rule both
   work — the eslint rule is preferred because it catches future
   regressions across all modules.

## Definition of done

- [ ] `TasksRepository` accepts a `ModuleDatabase`, not a raw prisma client.
- [ ] Tasks tables live in `krwn_tasks_<stateIdPrefix>` schemas (verified
      by a DB integration test when `TEST_DATABASE_URL` is set).
- [ ] No code under `src/modules/tasks/` imports `@/lib/prisma` directly.
- [ ] Regression test or eslint rule enforces this going forward.
- [ ] Existing S1.1 unit tests still pass (update mocks as needed).
- [ ] `docs/MODULE_GUIDE.md` gains a "Database access" section documenting
      the sandbox-only rule.
- [ ] CI green including integration job.

## Don't do

- Don't generalize the sandbox contract itself. If it's insufficient for
  tasks, open an issue and flag it in the PR rather than extending
  `ModuleContext` inline — that's a SDK change that belongs in its own PR.
- Don't retrofit `core.chat`, `core.wallet`, or `core.governance` into the
  sandbox in this PR. Those migrations are out of scope; track them as
  separate tasks for the parking lot.
- Don't drop the `TasksRepository` class abstraction. The point is to
  narrow its prisma dependency, not remove the repo pattern.

## Pitfalls

- Prisma client generation: if the task tables already exist in the public
  schema from yesterday's work, you'll need a migration that moves them.
  Look at `prisma/migrations/` for prior schema-move patterns
  (`20260420160000_node_subscriptions_wallet_fines` is an example of a
  multi-table migration).
- `ctx.db.transaction` wraps Prisma's `$transaction`. Make sure nested
  calls don't escape the scoped schema — the helper should set
  `search_path` on the transaction itself.
- Existing e2e `e2e/smoke.spec.ts` might hit tasks routes; re-run e2e
  locally with `npm run build && npm run test:e2e` before opening the PR.
