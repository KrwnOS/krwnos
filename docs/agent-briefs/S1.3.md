# S1.3 — Document first-party vs third-party module tiers

**Stage:** 1 · **Size:** S (2–4 hours) · **Blocked by:** S1.2 ·
**Blocks:** S1.4

**Branch base:** `cursor/mission-whitepaper-obligations` (NOT `main`).
**PR target:** `cursor/mission-whitepaper-obligations`.
**Your branch name:** `cursor/s1-3-module-tiers-docs`.

## Goal

Document the two-tier module model (first-party bundled vs. third-party
marketplace-installed) in `docs/MODULE_GUIDE.md`. Ensure `core.tasks`
follows the first-party convention (uses the Prisma client directly). If a
prior subagent attempt partially refactored `src/modules/tasks/repo.ts` to
`ModuleDatabase`, revert it. Defer strict sandbox enforcement for
first-party modules indefinitely — the sandbox is for untrusted code
installed through Stage 3's marketplace.

## Why this scope (and not the original)

The original S1.3 brief asked for `core.tasks` to be wired through the
`ModuleDatabase` sandbox. A subagent attempted it and hit a real
architectural problem: `ModuleDatabase` gives modules raw SQL with
`search_path` scoped to their own schema, but tasks legitimately needs to
reference `User` and `VerticalNode` in the public schema. The strict
sandbox doesn't have a clean cross-schema read story yet.

Decision made 2026-04-23: first-party modules bundled in the monorepo
(`core.chat`, `core.wallet`, `core.governance`, `core.tasks`) are
trusted code and use the Prisma client directly. The sandbox is
structurally the right enforcement boundary for **third-party** modules
installed via `krwn module install` in Stage 3. That's where it lives.

   If `src/modules/tasks/repo.ts` no longer uses the Prisma client

2. **Add a "Module tiers" section to `docs/MODULE_GUIDE.md`.** Cover:
   - **First-party (bundled) modules.** Live in `src/modules/*`,
     registered in `src/modules/index.ts`. Ship in the monorepo under
     your review. Use the Prisma client directly
     (`import prisma from "@/lib/prisma"`). Permission checks go through
     `permissionsEngine.can(...)`. Events go through `ctx.bus`. Examples:
     `core.chat`, `core.wallet`, `core.governance`, `core.tasks`.
   - **Third-party (marketplace) modules.** Installed via
     `krwn module install <file>.krwn` in Stage 3. Sandboxed via
     `ModuleContext.db` (raw SQL scoped to `krwn_<slug>_<stateIdPrefix>`
     schema), secrets via `ctx.secrets.get()`, no direct Prisma access,
     no cross-schema reads. Enforcement happens at install time through
     signature verification + manifest validation.
3. **Add a short "Database access" section** right after, making the
   per-tier pattern explicit. One paragraph each, with a code snippet
   showing the canonical import for each.
4. **Update `docs/ROADMAP.md`.** In §5 (Horizon 3 — Экосистема модулей),
   add a line under the sandboxing entry noting: "Strict sandbox applies
   to third-party modules installed via `krwn module install` (see Stage
   2/3). First-party bundled modules use Prisma directly — documented in
   `docs/MODULE_GUIDE.md` §Module tiers." Bump the header "Обновлено" date.
5. **No code changes to the sandbox kernel.** `src/core/module-sandbox.ts`,
   `packages/sdk/src/module-contract.ts`, and their tests stay as-is.
6. **No new eslint rule.** The original brief proposed one to forbid
   `@/lib/prisma` imports from modules — that assumption is wrong under
   the two-tier model. First-party modules legitimately import it. If
   the user wants a rule, it belongs in Stage 3 where it can check
   installed-via-marketplace modules specifically.
