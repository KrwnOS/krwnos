# S1.5 — Update `ROADMAP.md` and merge the in-flight branch

**Stage:** 1 (final) · **Size:** S (half day) · **Blocked by:** S1.4 ·
**Blocks:** S2.1 and all later stages

## Goal

Close the `cursor/mission-whitepaper-obligations` branch cleanly. Every
ROADMAP §5 entry for the Stage 1 work gets a commit/PR reference, the
branch is merged to `main`, and the repo is ready for Stage 2.

## Why this matters

ROADMAP.md §0 explicitly requires that any scope change land in the same
PR. Right now `core.tasks`, DB sandboxing, and `ctx.secrets` are all
marked ✅ 2026-04-22 with no commit ref, because the branch hasn't
merged. Stage 2 (signed modules) cannot start on a branch this long-lived
without Sprint-1-style conflicts in `package-lock.json`, `schema.prisma`,
and the SDK. Landing is the gate.

## Starting state

- Branch `cursor/mission-whitepaper-obligations` with S1.1 through S1.4
  merged into it.
- `docs/ROADMAP.md §5` contains three entries marked ✅ 2026-04-22 with
  `(#—)` placeholders in the commit-ref slot.
- `CHANGELOG.md` and `docs/ROADMAP.md §0` already reflect the S1.4
  reconciliation.

## Do this

1. Rebase `cursor/mission-whitepaper-obligations` on the latest `main`.
   If anything else merged while Stage 1 was in progress, resolve
   conflicts; this is why the merge-train rule exists.
2. Open the PR. In the description, link every ROADMAP §5 and §9 entry
   this branch closes, and list the S1.\* brief IDs for traceability.
3. After CI is green and review passes, squash-merge to `main`. Take the
   resulting commit SHA.
4. In a follow-up tiny commit on `main` (or as part of the squash if
   possible), fill in the `(#—)` placeholders in `ROADMAP.md §5` entries
   with the real commit/PR URL. Also update `ROADMAP.md §1` "статус по
   фазам" table if any phase-level state changed (Phase 5 / Horizon 3 is
   still `⏳ active`; don't change unless genuinely closed).
5. Bump `ROADMAP.md` header "Обновлено" date.
6. Delete the branch locally and on the remote.
7. Tag if appropriate. If this is the first time Horizon 3's
   sandboxing pillar is fully landed, consider `v0.1.1` or similar — but
   don't tag unless you've also reviewed `CHANGELOG.md [Unreleased]`
   for user-facing completeness.

## Definition of done

- [ ] Branch merged to `main`, CI green.
- [ ] `ROADMAP.md §5` entries (`@krwnos/sdk`, sandboxing, `ctx.secrets`,
      `core.tasks`) all have real commit/PR refs.
- [ ] `ROADMAP.md` header date updated.
- [ ] Branch deleted on remote.
- [ ] `git log --oneline -5` shows a clean squash commit on `main`.
- [ ] `git status` clean on `main`, no unmerged work left behind.

## Don't do

- Don't open Stage 2 tasks in the same PR or on the same branch. S2.1
  gets a fresh branch after this merges.
- Don't `git push --force` to `main`. If the rebase gets ugly, resolve
  it on the feature branch and re-PR.
- Don't skip the ROADMAP update step. The §0 contract is load-bearing;
  future agents rely on it to know what's actually landed.

## Pitfalls

- If CI reveals that Stage 1 work missed something (flaky e2e, coverage
  edge case), you have two options: patch on the branch and re-request
  review, or open a follow-up. Prefer the first for anything essential,
  the second for polish. Do not merge red CI to `main`.
- Tagging is optional but valuable. If you tag, write the release notes
  from `CHANGELOG.md [Unreleased]`, then move those entries under a new
  `## [0.1.1] - 2026-04-YY` heading and reset `[Unreleased]` to empty
  with the standard subheadings.
