# S2.4 — `krwn module install` verifies signature

**Stage:** 2 · **Size:** M (2–3 days) · **Blocked by:** S2.2 and S2.3 ·
**Blocks:** S2.5

## Goal

Extend the existing `krwn module install` path so that installing a
`.krwn` archive (a) resolves the publisher via `@krwnos/sdk`'s registry,
(b) verifies the Ed25519 signature against the archive bytes, (c)
re-runs manifest validation, (d) refuses to touch the database on any
failure. Every failure emits a distinct error code.

## Why this matters

This is the security gate. Everything upstream (spec, pack/sign,
registry) exists so this verifier can be simple, auditable, and
bulletproof. A bug here means a malicious archive could run migrations
or register a module under a spoofed identity. There are no second
chances on this code path — it deserves the most test coverage in the
whole codebase.

## Starting state

- `cli/src/commands/module/install.ts` (or wherever install currently
  lives — check `cli/src/commands/module/` first) installs an unsigned
  archive.
- `packages/sdk/src/manifest.ts` — `validateKrwnModuleManifest`.
- `packages/sdk/src/publishers.ts` (from S2.3) — publisher lookup.
- `docs/MODULE_PACKAGING.md` — error taxonomy to implement verbatim.
- `src/app/api/cli/modules/route.ts` — server-side route that the CLI
  talks to when installing into a running State. This may need updates
  too: the server should re-verify the signature, not trust the CLI.
- `fixtures/modules/example-good/` — starting fixture.

## Do this

1. Implement the verify pipeline as a shared helper in `@krwnos/sdk`
   (so both CLI and server side can use it):
   `verifyKrwnArchive({ archivePath, sigPath }): Promise<VerifiedModule>`.
   Steps in order: load archive bytes; load `.sig`; extract
   `publisherKeyId` from the envelope; resolve via
   `resolvePublisher`; if unknown → `KRWN_PKG_UNKNOWN_PUBLISHER`;
   verify signature with `crypto.verify`; if invalid →
   `KRWN_PKG_BAD_SIGNATURE`; extract manifest from archive; validate
   → `KRWN_PKG_MANIFEST_INVALID` on failure; check
   `minPlatformVersion` → `KRWN_PKG_VERSION_MISMATCH` on failure.
2. Update the CLI `module install` command to call the helper before
   proceeding. If verification fails, exit non-zero with the error code
   in the message and abort before any DB operation.
3. Update the server route (`src/app/api/cli/modules/route.ts`) to
   **re-run** verification server-side. The CLI is not trusted — it
   could be anything. The server verifies the bytes it received.
4. Migrations only run after verification succeeds. Any failure leaves
   the State untouched.
5. Add a server-side integration test at
   `src/app/api/cli/__tests__/modules-install.integration.test.ts`
   covering: happy path (signed fixture installs and appears in
   `InstalledModule`); tampered archive rejected with
   `KRWN_PKG_BAD_SIGNATURE`; unknown publisher rejected; bad manifest
   rejected; no DB side effects on any failure path.
6. Add fixtures: `fixtures/modules/example-signed/` (properly packed +
   signed with the core key), `fixtures/modules/example-tampered/`
   (the archive with a byte flipped after signing). A fixture builder
   script under `fixtures/modules/build.mjs` that generates these on
   demand — do not commit the `.krwn` archives if they change per
   build, but commit the script.
7. Update `docs/CLI.md` with the new install behavior and error codes.
8. Update `ROADMAP.md §5` Signed Modules line: mark `[x]` with commit
   ref.

## Definition of done

- [ ] `verifyKrwnArchive` helper lives in `@krwnos/sdk` with full
      coverage.
- [ ] CLI `module install` verifies before acting.
- [ ] Server route verifies independently.
- [ ] All four negative tests fail cleanly with distinct error codes

- [ ] Happy path: fixture installs, `InstalledModule` row present,

- [ ] `docs/CLI.md` and `docs/MODULE_PACKAGING.md` updated.
- [ ] Coverage ≥ 90 % on the verifier (this is security-critical; the

- [ ] ROADMAP updated.

## Don't do

- Don't trust the CLI's self-verification to justify skipping the
  server check. Two-sided verification is not redundant; it's
  defense-in-depth.
- Don't allow `--skip-verify` or any bypass flag. If someone needs to
  install an unsigned module during development, they use the existing
  path to register it via code (`registry.register()`) — the install
  flow is for signed distribution only.
- Don't decompress the archive into the running process's memory
  without size limits. Enforce the caps from the S2.1 spec
  (compressed, uncompressed, file count) before reading further.
- Don't log the signature bytes, the archive hash, or the public key
  in a way that leaks to user-visible output beyond what's needed for
  error messages.

## Pitfalls

- `crypto.verify('ed25519', data, publicKey, signature)` in Node
  expects `publicKey` as a `KeyObject` or PEM buffer. Double-check the
  format flowing out of `publishers.json`.
- Large-archive DoS: set an absolute cap on decompressed size. A 1 GB
  gzipped archive can decompress to 100 GB. Use a streaming
  decompressor and abort past the cap.
- Path traversal in tar entries: reject any entry whose extracted path
  escapes the module root (`..`, absolute paths, symlinks outside the
  tree). The `tar` package has options for this — use them.
- If you can't recreate the exact canonicalization the packer used,
  the happy path will fail. That's a packer bug, not a verifier bug —
  but investigate both sides before declaring either one correct. The
  round-trip test from S2.2 is the canary.
