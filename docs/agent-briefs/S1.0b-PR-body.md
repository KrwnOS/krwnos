# fix(contract): reconcile @krwnos/sdk and core.tasks typecheck drift

Closes S1.0b. Makes `npm run typecheck` clean on
`cursor/mission-whitepaper-obligations` so that S1.1 onwards can
proceed. Branch base: `cursor/mission-whitepaper-obligations`
(not `main`).

## Summary

`core.tasks` was generated against an imagined shape of `@krwnos/sdk`
and `PermissionsEngine` that does not exist on this branch. Typecheck
was failing with ~40 errors, most inside `src/modules/tasks/`. This PR
walks through the 8 decisions in the brief, extending the SDK where a
module's assumption reflects a legitimate future direction and fixing
the module where the mismatch was a mistake.

Error count monotonically decreased after each commit:
40 → 26 → 25 → 23 → 22 → 14 → 6 → 5 → 1 → 0.

## Deviations

- **Decision 4** — the brief guessed `CanInput = { stateId, userId, nodeId? }`.
The real `PermissionCheckInput` is `{ stateId, userId, isOwner, snapshot }`
and `can()` has no `nodeId` parameter (node scoping uses
`isMemberOfNodeOrAncestor`). Service methods now take
`(ctx: ModuleContext, access: TasksAccessContext, ...)` where
`TasksAccessContext = { isOwner, snapshot }`. This mirrors how
chat / governance / wallet assemble their access context. Engine
public signature untouched, per brief rule. `TasksAccessContext`
is explicitly temporary — S1.2 is expected to retire it in favour
of a shared helper.
- **Decision 5** — `getAuthenticatedContext(req)` returns
`{ ctx, access }`, not the bare `ModuleContext` hinted at in the
brief. Rationale: Decision 4 moved the service to a `(ctx, access)`
call shape, so the stub has to hand both back. TODO comment in
`_context.ts` points at S1.2 for extraction.
- **9th, implicit** — `src/modules/tasks/index.ts` also used a
nonexistent `manifest` / `bootstrap` shape, not called out in the
brief. Rewritten to the real `KrwnModule` contract
(`slug`, `name`, `version`, `init`, `getWidget`, `getSettings`).
Commit `cbaeea1`.

## SDK surface additions (brief DoD)

Documented in `docs/MODULE_GUIDE.md`:

- `ModuleAuth = { userId: string }`
- `ModuleContext.auth: ModuleAuth | null` (null ⇒ system-initiated
call: background job, hook, event)
- `KrwnError(message, code)` — error class for module → route code
mapping (e.g. `"FORBIDDEN"` → HTTP 403).

No existing first-party module was broken by the extension — wallet /
governance / chat build `ModuleContext` via `createSandboxedContext`,
which now populates `auth` automatically.
