## Goal

Add vitest unit tests for the `core.tasks` module (service + repository) so
the project's 70 % core-coverage gate doesn't regress when the in-flight
branch merges.

## Why this matters

`core.tasks` landed on 2026-04-22 as a "Trello-like Kanban" module and
currently has **zero tests**. Every other module (`chat`, `governance`,
`wallet`) has at least service-level unit tests. Without these, the branch
`cursor/mission-whitepaper-obligations` cannot merge cleanly because CI
coverage thresholds will fail.

## Do this

   - `getAccessibleBoards`: unauthorized (no `ctx.auth`) throws
     `UNAUTHORIZED`; missing `tasks.read` throws `FORBIDDEN`; node-scoped

## Don't do

- Don't add integration tests that require a live Postgres. Those belong in
  a `.integration.test.ts` sibling file, gated by `TEST_DATABASE_URL`. S1.3
  will touch the real-DB side via the sandbox contract; save it for that.
- Don't refactor `service.ts` or `repo.ts` behavior just to make them easier
  to test. If a test is hard to write, note it in the PR description —
  S1.2/S1.3 will do the refactor with context.
- Don't import `@/lib/prisma` directly in tests. Mock the repo or the prisma
  client passed into it.

## Pitfalls

- `TasksRepository` currently takes the global prisma singleton. Pass a mock
  client in tests rather than monkey-patching the module. S1.3 will formalize
  this through `ModuleContext.db`.
- `permissionsEngine` is a singleton exported from `src/core/permissions-engine.ts`.
  Either stub it with `vi.mock`, or construct a fresh one per test — the
  existing governance tests show both approaches.
